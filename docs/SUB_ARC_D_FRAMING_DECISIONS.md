## TL;DR

- **Mode 2 · Conversational** (D.v1 · ships FIRST by default) — engineer chats with Canvas AI Assist normally; individual messages can emit structured action proposals inline; engineer reviews + applies via preview modal. Use case: everyday canvas refinement, mid-session edits. Effort: ~1-2 weeks.

## Q1 · UI Surface for Mode 1 · LOCKED

**Decision**: Dedicated **"Workshop Notes" overlay** invoked via `Cmd+Shift+N` (primary keyboard affordance) + footer button `"📝 Workshop notes"` (discoverable backup). NOT a 6th main tab. NOT inside Canvas Chat overlay's tab strip. NOT a new topbar button.

**Reasoning** (grounded in CH26 + CH28 + CH32 + R20.13):

| Option | Verdict | Why |
|---|---|---|
| 6th main tab | ❌ | Dilutes the 5-tab workshop progression (Context → Current → Desired → Gaps → Reporting). Note-taking isn't a workshop STAGE; it's a CAPTURE mode that runs alongside any stage. |
| Inside Canvas Chat overlay's tab strip | ❌ | Semantically confuses what's AI-active (chat round-trips) vs AI-deferred (note batch-processing). Q7's "no proposal-generating round-trips during note-entry" makes it conceptually distinct from the chat tab. |
| New topbar button | ❌ | Violates CH26 single-AI-affordance-on-topbar contract. |
| **Dedicated NEW overlay (Cmd+Shift+N)** | ✅ | Matches existing AI Assist overlay pattern (R20.13). Doesn't disturb tabs. Doesn't conflict with CH26 (topbar stays single-AI). Keyboard shortcut is the live-workshop ergonomic primary affordance (engineer can open + close fast without breaking customer conversation flow). |
| Footer "📝 Workshop notes" button | ✅ as backup | Footer already hosts non-tab actions (Save to file · Open file · Import data · Load demo). Adding "📝 Workshop notes" follows precedent without overloading topbar. |

**Overlay shape** (Mode 1 surface implementation contract):

- **Header**: `"Workshop notes · <customer name> · <today's date>"` (context-aware; reads from `engagement.customer.name` and current date)
- **Body**: dark distraction-free editor surface (matches Canvas Chat overlay's transcript dark surface · `#0D1117` bg) covering ~70% of viewport
- **Editor**: markdown-rendered live as engineer types (consistent with how Canvas Chat renders assistant turns)
- **Footer**: `[Exit & process with AI]` (primary action · blue Dell-style button) + `[Save draft & close]` (secondary · no AI call)
- **Persistence**: localStorage auto-save every keystroke under `workshopNotesDraft_v1` key (mid-workshop browser-close protection; survives reload)
- **Exit & process flow**: real anthropic LLM call (per `feedback_no_mocks.md`) → returns `(markdown_notes_file_contents, action_proposal_list)` → preview modal opens (extends `ImportPreviewModal` from §S47) → engineer reviews + applies populations selectively → markdown file offered for download via `[Save discovery notes]` button

**NOT in this overlay**: chat round-trips · grounding verifier · proposal generation during note entry · typeahead OR autocomplete generated by LLM calls. (See Q7 for what IS allowed.)

## Q2 · `flip-disposition` Action Kind · DEFERRED to v1.5

**Decision**: Sub-arc D v1 ships 4 additive action kinds. `flip-disposition` moves to v1.5 alongside delete/hide/archive once cascade-confirmation UX patterns land.

**v1 action kinds (locked)**:
- `add-driver`
- `add-instance-current`
- `add-instance-desired`
- `close-gap`

**Deferred to v1.5+**:
- `flip-disposition` (cascade risk · originId chains can break · linked gaps may need re-evaluation)
- `delete-driver` · `delete-instance` · `delete-gap` (destructive · needs undo confirmation UX)
- `hide-environment` (lifecycle nuance · soft-delete pattern)
- `archive-engagement` (cross-cutting · needs multi-engagement support that ships v3.1+)

**Why defer flip-disposition** (beyond the v1 cut):
- Mutates an existing instance's lifecycle role rather than adding a new entity
- Flipping Replace → Consolidate breaks the 1-to-1 origin pairing semantics codified in `instance.originId`
- May cascade to linked gaps (per §S25 disposition lifecycle table); chat would need cascade-awareness
- The 4 additive kinds have natural rollback (remove the added entity); flip-disposition needs explicit revert UX

## Q4 · Sequencing · Mode 2 FIRST (with toggle clause)

**Default decision**: Mode 2 (Conversational · D.v1) ships first; Mode 1 (Workshop Notes · D.v2) ships second.

**Toggle clause**: Sequencing flips to Mode 1 first if a live customer workshop is scheduled within ~3 weeks of rc.10 start. Otherwise Mode 2 first.

**Why Mode 2 first by default** (engineering-risk argument):

- **Thinner**: chat infrastructure exists, preview modal extends `ImportPreviewModal` from §S47, action proposals layer onto existing chat turns
- **Validates the shared infrastructure** (action-proposal schema · preview modal extension · commit functions · provenance enum) in a LOW-COUPLING context BEFORE Mode 1 takes a dependency on it
- **Matches our cadence** (Sub-arc A → B → C went smaller-and-more-targeted first; the eval harness landed before persona examples landed before the schema-truthful rule + docs)
- **Less risk** of "Mode 1 was building on shifting infrastructure" mid-arc
- **Effort**: ~1-2 weeks

**Why Mode 1 second**:

- Mode 1 is THICKER: new UI surface · new state · new UX patterns for entry/exit/persistence · new file-save flow
- Mode 1 USES the validated infrastructure from Mode 2's ship (lower-risk Mode 1 build)
- **Effort on validated infra**: ~2-3 weeks

**Why the toggle clause matters**:

- The product-priority for Mode 1 is the live-workshop ergonomic painpoint
- If a real workshop is happening in 3 weeks, shipping Mode 2 first (1-2 weeks) + then Mode 1 (2-3 weeks) = 3-5 weeks total · Mode 1 misses the workshop
- Flipping to Mode 1 first (2-3 weeks · uses-uninvalidated-infra · slightly higher risk) means Mode 1 is ready in time
- Outside the 3-week window, engineering-risk-minimizing sequencing dominates

## Q5 · Rubric Integration · ORTHOGONAL EVALS (separate harnesses)

**Decision**: Sub-arc D's action-correctness rubric is a SEPARATE harness from Sub-arc A's chat-quality rubric. Two independent evals; two independent baselines; two independent measurement bars.

**Rationale**:

- Sub-arc A's 5 dims (grounded · complete · useful · honest · concise) measure chat TEXT quality
- Sub-arc D's 5 dims (action-kind correctness · target-state · payload · confidence · restraint) measure action CORRECTNESS
- The two axes are conceptually orthogonal (text quality vs action correctness)
- In Mode 2, a chat turn can have BOTH a chat-text component AND an action-proposal component; layering them into a 10-dim case adds judge complexity, parse-error risk, and obscures which axis is improving/regressing

**Implementation**:

- New harness file: `tests/aiEvals/actionCorrectnessHarness.js` (or similar; final naming at eval-build time)
- New golden set: `tests/aiEvals/actionGoldenSet.js` (D-specific cases · separate from existing 25-case chat-quality set)
- New rubric: `tests/aiEvals/actionRubric.js` (5 dims · same 0-2 scoring · same 7/10 pass threshold)
- New judge prompt: `tests/aiEvals/actionJudgePrompt.js`
- Browser-runnable via `window.runCanvasAiEvals({ harness: "action-correctness" })`
- Independent baseline: `tests/aiEvals/baseline-action-<timestamp>.json` (timestamped; canonical at `tests/aiEvals/baseline-action.json` per the rc.9 baseline.json convention)

**Cross-cutting analysis**: when a Mode 2 case has BOTH a chat-text component AND an action-proposal component, both harnesses run; results stored separately. Post-capture correlation analysis can answer "do cases with high chat-quality also have high action-correctness?" without coupling the rubrics.

## Q6 · Rule 4 Evolution · AMENDED

**Current Layer 1 Rule 4** (verbatim from `services/systemPromptAssembler.js`):

> "You may propose changes (rename, re-classify, re-link) but you may NOT mutate the canvas. End every proposal with 'click apply if you want me to open that view for you.'"

**Proposed amendment** (rc.10 Sub-arc D · constitutional touch at preamble time · NOT applied yet in this framing doc):

> "You may propose changes (rename, re-classify, re-link, add-instance, close-gap, etc.) but you may NOT mutate the canvas. When the proposal matches a known action-kind (per the action-proposal schema in SPEC §S20.X NEW), emit a structured action object; the UI renders the Apply button automatically. For free-text proposals without a structured-action shape (rename ideas, qualitative re-classifications), end with 'click apply if you want me to open that view for you' as a navigation prompt."

**Style note**: the rewrite avoids em dashes inside the rule body. Layer 1 prompt text is baked into the chat's system prompt and read by the LLM every turn. Em dashes in the rule body are a stylistic LLM-tell that can correlate with LLM-formal-tone output; using semicolons + period-separated clauses keeps the rule body neutral. (This is a defensive style choice; not formally locked in memory anchors but applied as a quiet convention for system-prompt text.)

**Why amend rather than leave as-is**:

- Sub-arc D OPERATIONALIZES the "propose then apply" pattern that was originally textual (chat says "you should rename X" + engineer clicks somewhere)
- Without amendment, the LLM might emit BOTH a textual proposal AND a structured action object for the same change (redundant + confusing)
- The amendment splits the rule: structured-action-shape proposals route through the schema + Apply button; free-text proposals keep the original textual-apply pattern

**Constitutional touch scope** (at preamble time):

- `services/systemPromptAssembler.js` Layer 1 Role section (Rule 4 rewrite)

## Q7 · No-Round-Trips Scope · SCOPED, not absolute

**Decision**: The "no LLM round-trips during note-taking" constraint in Mode 1 is SCOPED, not absolute.

**OK during note-entry** (cached-locals only · no network · no LLM inference):

- Markdown rendering of the engineer's typed notes (live as they type)
- Typeahead against the cached concept dictionary (62 entries · already in Layer 3 of the system prompt · available client-side)
- Cached workflow TOC autocomplete (16 workflows · already in Layer 5 · client-side)
- Inline schema-shape validation (real-time as engineer types · structural-only)
- localStorage auto-save every keystroke

**NOT OK during note-entry** (would require network + LLM inference):

- Any anthropic / external-LLM round-trip
- Grounding verifier calls
- Proposal-generation invocations
- Real-time chat-style assistance

**Why scoped**:

- The constraint is about deferring LLM INFERENCE-LATENCY (which is what would break the customer-conversation flow)
- Cached-local UX intelligence (typeahead from already-loaded data) is fast and engineer-helpful
- Distinguishing inference-latency-causing actions from data-lookup actions clarifies what's allowed

**Behavioral envelope for the Exit & process call**:

- ONE round-trip when engineer clicks `[Exit & process with AI]`
- The accumulated notes are sent as a single payload
- LLM returns `(markdown_notes, action_proposal_list)` in one response
- Preview modal opens; further LLM calls only happen if engineer triggers something else (not on idle, not on hover, not on tile selection)

## Action-Proposal Schema · First Cut

This is a FIRST DRAFT for orientation; the canonical schema lands in `schema/actionProposal.js` (NEW · created at preamble time) with full Zod validation. The shape below is illustrative.

```js
// schema/actionProposal.js (NEW · created at constitutional preamble time)
//
// Each proposal the chat emits is one ActionProposal object. Mode 1
// batch-process returns ActionProposal[]; Mode 2 inline returns
// ActionProposal[] alongside the chat text turn.

ActionProposalSchema = z.object({
  // Discriminator for which canonical commit function this routes through.
  // v1 enum (4 kinds); flip-disposition + destructive kinds deferred.
  kind: z.enum([
    "add-driver",
    "add-instance-current",
    "add-instance-desired",
    "close-gap"
  ]),

  // For add-instance-* only. Identifies which state the new tile lands in.
  // Tab 2 (current) or Tab 3 (desired).
  targetState: z.enum(["current", "desired"]).optional(),

  // Polymorphic per-kind payload. Shape varies by `kind`.
  // - add-driver:           { businessDriverId, priority?, outcomes? }
  // - add-instance-current: { layerId, environmentId, label, vendor?,
  //                           vendorGroup?, criticality? }
  // - add-instance-desired: { layerId, environmentId, label, disposition,
  //                           originId?, vendor?, vendorGroup? }
  // - close-gap:            { gapId, status: "closed", closeReason? }
  payload: z.record(z.unknown()),

  // Chat-emitted confidence about this specific proposal. Lets the
  // engineer triage (high → apply with quick review; low → scrutinize
  // before applying).
  confidence: z.enum(["high", "medium", "low"]),

  // Chat-emitted natural-language rationale. The engineer reads this
  // in the preview modal to decide whether the proposal makes sense.
  // E.g.: "The customer mentioned ransomware concerns; this driver
  // captures that thread of the engagement."
  rationale: z.string().min(1),

  // Which mode emitted this proposal. Routes to the corresponding
  // aiTag.kind value at apply time:
  //   "discovery-note" → aiTag.kind = "discovery-note" → "Note" chip
  //   "ai-proposal"    → aiTag.kind = "ai-proposal"    → "Prop" chip
  source: z.enum(["discovery-note", "ai-proposal"])
}).strict();

// Convenience type for collections.
ActionProposalListSchema = z.array(ActionProposalSchema);
```

**Cross-references for the schema build phase**:

- Existing `aiTag` shape: `schema/instance.js` (extended in §S25 amendment + §S47.9 for `kind` discriminator)
- Existing canonical commit functions: `state/collections/driverActions.js` · `state/collections/instanceActions.js` · `state/collections/gapActions.js`
- Existing preview modal: `ui/components/ImportPreviewModal.js` (Path B Import · §S47)
- The new schema would land at `schema/actionProposal.js` (matches existing `schema/customer.js` / `schema/engagement.js` / etc. pattern)

### A2 · Mode 1 UX detailed spec · dual-pane overlay

The Mode 1 surface implementation contract (extends Q1 above with concrete UX detail captured from user direction):

**Layout · two-pane vertical split**:

**Upper pane · Processed notes (read-mostly, AI-formatted)**:
- AI-formatted output of pushed notes
- Structured-by-topic sections — the AI groups the engineer's raw bullets under canonical headings: `## Customer concerns` · `## Drivers identified` · `## Current state captured` · `## Desired state directions` · `## Gaps proposed` · `## Action items / follow-ups`. This grouping matches the canvas data model and makes the discovery notes file useful for next-session prep
- Each processed note item has an **expansion icon** revealing canvas mapping details (which drivers, environments, instances, gaps it maps to, with confidence)
- Rich-text markdown rendering
- Auto-scrolls to latest content on each push
- Read-mostly — engineer can scroll/expand/select but doesn't edit directly (the source of truth is the lower pane's raw bullets; the upper pane is regenerated from them)

**Lower pane · Note capture (always-visible, input-focused)**:
- Auto-bulletpoint editor — `Enter` creates a new bullet · `Tab` indents · `Shift+Tab` outdents
- Plain structured bullets — no autocomplete, no LLM round-trips during typing (per Q7 lock)
- Always-visible during the session — push events do NOT disturb this pane
- Auto-saved to localStorage every keystroke under `workshopNotesDraft_v1` key

**Pane size**: 60% upper / 40% lower default; engineer can drag the divider to resize (persisted in localStorage).

### A3 · Push behavior · DELTA-ONLY by default + explicit "Re-evaluate all" button

**Locked**: each push processes ONLY the new notes since the last push (delta-only · append semantics). Earlier mappings stay stable.

**Why**: matches the user's word "amend"; cheap + fast + predictable for long workshops (2-4 hours). Re-processing the whole 24h block on every push would balloon LLM input tokens + latency as the workshop progresses.

**Escape hatch**: an explicit **`[Re-evaluate all]` button** in the upper-pane toolbar lets the engineer re-process the entire current-day block when they want the AI to revise earlier mappings based on later context. Power-user feature, not the default behavior.

**Push triggers**:
- Button `[Push notes to AI]` in the lower pane
- Keyboard shortcut `Cmd+Enter` (or `Ctrl+Enter` on Windows/Linux)
- Both fire the same action

**On push**:
- DELTA only: new bullets since last push are sent to the LLM with the existing formatted-notes context as prior turns
- LLM returns formatted addition + new canvas mappings → appended to upper pane (no replacement of earlier formatted text)
- Lower pane unchanged

**On `[Re-evaluate all]`** (explicit):
- Full current-day block re-processed
- Upper pane regenerated from scratch
- Earlier mappings may shift → revised mappings get a small `[revised]` indicator next to the mapping row in the expansion view
- Engineer can accept or override per-mapping

### A4 · Mapping confidence display · text labels + subtle color tinting

**Locked style** for confidence in the expansion-view mapping rows:

- **Text label** in the row: `HIGH` · `MEDIUM` · `LOW`
- **Subtle color tinting** on the row background:
  - HIGH: light green tint (`#E8F5E9`-ish · slightly green-tinted neutral)
  - MEDIUM: light amber tint (`#FFF8E1`-ish)
  - LOW: light gray tint (`#F5F5F5`-ish · don't use red · LOW is "I'm uncertain", not "this is wrong")
- **NO numeric percent** — avoids false precision (the LLM doesn't actually emit a calibrated 0-100 score; high/medium/low maps to the chat's qualitative emit per the action-proposal schema)

**Why**: engineers triage faster with discrete labels + at-a-glance color than they do with numbers. Red would imply "rejected/error"; gray is "low confidence, review before applying" — preserves engineer agency.

### A5 · Mapping review · per-row + bulk-apply (both available)

**Per-mapping review** (primary flow):
- Each processed note has an expansion icon
- Clicking expands to show: identified canvas data points (drivers, environments, current/desired instances, gaps) + confidence + per-mapping `[Apply]` / `[Reject]` buttons
- Applied mappings carry `aiTag.kind: "discovery-note"` provenance (per Q3 lock)

**Bulk-apply** (power flow):
- Toolbar button `[Review all mappings]` in the upper pane opens a sidebar OR modal listing every proposed mapping in a flat list with checkboxes
- Engineer can multi-select + `[Apply selected]` in one action
- Useful when engineer wants to triage 20+ proposals from a long workshop fast

### A6 · 24-hour divider · calendar-day boundary in engineer's local timezone

**Locked**: notes file accumulates across sessions. Each calendar-day boundary (midnight in engineer's local timezone) inserts a new divider with metadata stamp on the next push:

```
─────────────────────────────────────
Session: 2026-05-15 · Customer: <customer name> · Engineer: <presales owner>
─────────────────────────────────────
```

- Within the same calendar day: amend continues (no new divider)
- After midnight on next push: new divider inserted, then the new processed content
