## Architecture decision committed to memory

`project_v3_no_file_migration_burden.md` (saved 2026-05-01) — based on user direction:

> "I don't want to spend too much worries and time migrating old saved files from 2 → 3 and from 3 → 2, as we don't have that need much at the moment and I don't want to influence the architecture of the v3 into something less efficient or more complicated to do this feature."

**Practical effect**:
- v2→v3 file migrator stays as-is on synthetic fixtures (V-MIG suite GREEN). Not widened for real-customer edge cases.
- No v3→v2 file exporter built.
- The original "real-customer `.canvas` migration smoke" GA gate is **dropped** from the v3.0.0 GA criteria.
- v3 schema design wins over back-compat. v3 is for v3.

The bridge module reflects this: it does NOT call the file-load migrator at runtime. It does an in-memory translation of the fields v3 AI surfaces actually consume, falling back to `createEmptyEngagement()` if validation fails.
